Phase 1 Roadmap — Storage Engine
Goal by the end: a working on-disk file format that can store, retrieve, update, and delete variable-length records across multiple pages — the foundation everything else (buffer pool, B+Tree, execution) sits on top of.
Step 1 — Slotted Page Design (concept + design doc, no code yet)

Learn: why flat byte arrays don't work for variable-length records (fragmentation, no way to locate records)
Learn: the slotted page pattern — a slot array growing from the front of the page, actual record data growing from the back, meeting in the middle
Deliverable: no code — write out (in your notes) the byte layout of your page: header fields, slot directory format, free space tracking
Reading: Database Internals Ch 3

Step 2 — Page Header & Slot Directory (code)

Redesign Page from {data: [u8;4096]} into something with a real header: page ID, number of slots, free space pointer, etc.
Implement the slot directory: each slot stores an (offset, length) pair pointing into the record area
Deliverable: insert_record(page, bytes) -> slot_id and get_record(page, slot_id) -> bytes working within a single page
Verify: insert several variable-length byte blobs into one page, retrieve each by slot id, confirm bytes match exactly

Step 3 — Deletion & Free Space Management

Implement delete_record(page, slot_id) — mark slot as deleted/tombstoned
Decide: do you compact/reclaim space immediately, or leave gaps and compact later (real DBs usually defer this)
Deliverable: delete a record, insert a new one, verify space reuse works without corrupting neighboring records
Verify: a test that inserts, deletes, and reinserts records and checks page integrity throughout

Step 4 — Record (Row) Format

Design how an actual row/tuple is serialized into bytes before it goes into a page (e.g. a row with an int column and a string column — how do you encode both into one byte blob, and decode them back?)
Deliverable: serialize_row(columns) -> Vec<u8> and deserialize_row(bytes) -> columns for a small fixed schema (e.g. i32 + variable-length string)
Verify: round-trip a few rows with different string lengths through serialize → insert into page → get from page → deserialize, confirm data matches

Step 5 — Multi-Page File Layer

Extend beyond one page: a file made of many pages, each identified by page number
Deliverable: DiskManager (or similar) with read_page(page_id) -> Page, write_page(page_id, page), allocate_page() -> page_id
This is your read_pages/write_pages/append_page code from Phase 0, generalized into a proper page-indexed interface instead of "read the whole file at once"
Verify: allocate several pages across a file, write different records into each, close and reopen the file, confirm everything reads back correctly

Step 6 — Integration Test / Phase 1 Checkpoint

Write one end-to-end test: create a file, allocate multiple pages, insert many rows (enough to span pages) via serialize_row → slotted page insert_record, delete some, reopen the file fresh, read everything back, verify correctness
This is the gate before Phase 2 (buffer pool) — buffer pool wraps this layer with caching, so it needs to be solid first
